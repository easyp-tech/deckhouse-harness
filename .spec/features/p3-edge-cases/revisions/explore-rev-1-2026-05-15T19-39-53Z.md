Закрыть оставшиеся 4 handler'а из roadmap (`@/Users/zergslaw/Projects/Sipki-Tech/deckhouse-mcp/ROADMAP.md`) и довести каталог Deckhouse MCP API до **43/43 handlers**. P3 — это нишевые / debugging операции, не критичные для типичных сценариев, но необходимые для полного покрытия:

### Deckhouse API specifics

- **`ModuleRelease` CRD** (`deckhouse.io/v1alpha1`): namespace-less, перечисляет версии модулей доступные через `ModuleSource`. Labels включают `module=<name>` для фильтрации.
- **Maintenance mode для ModuleConfig**: реализуется через `spec.maintenance: NoOperations` (или похожее поле в API, точное название — `[ASSUMPTION: Maintenance mode реализован через поле "maintenance" в spec, нужно подтвердить из CRD schema]`).

## Options Considered

### Option A: Single-batch implementation (recommended)

Все 4 handler'а в одной фазе разработки (одна последовательность RED→GREEN→CODE).

- **Pros:** P3 = всего 4 handler'а, не требует разбиения; единый CHANGELOG entry; один merge commit; минимальный coordination overhead
- **Cons:** Если возникает блокер на одном handler'е — задерживает остальные
- **Complexity:** Low. ~3-4 часа реализации + ~2 часа тестов (по аналогии с P2 Batch 3 — 4 handler'а SourcesAPI заняли ~6 часов wall-clock)

### Option B: Two batches by domain

Batch 1: B6 (ModuleConfig patch) + D13 (новый CRD NodeGroupConfiguration). Batch 2: F3 (DeleteModuleSource) + F6 (новый CRD ModuleRelease).

- **Pros:** Domain-grouping; легче review per batch
- **Cons:** Overhead — 2 PR, 2 CHANGELOG entries; не оправдано для 4 handler'ов
- **Complexity:** Same total time + 30 min coordination

### Option C: One-by-one (sequential)

Каждый handler — отдельная commit'ом / phase.

- **Pros:** Максимальная изоляция при review
- **Cons:** Чрезмерный overhead для edge cases; 4 раза проходить generate/lint/build
- **Complexity:** +1-2 часа из-за переключений контекста

## Constraints & Risks

- **Maintenance mode API uncertainty** — точное имя поля в `ModuleConfig` (`spec.maintenance` vs `spec.suspended` vs другое) нужно подтвердить из Deckhouse docs/CRD schema перед requirements phase. Открытый вопрос.
- **NodeGroupConfiguration security** — handler позволяет inject bash-кода, выполняемого на нодах с root правами. Требуется RBAC только для авторизованных операторов; в нашем случае ServiceAccount `deckhouse-mcp` уже cluster-scoped, дополнительных ограничений не накладываем (MCP server is admin-level by design).
- **ModuleRelease scope** — CRD может иметь сотни записей (по модулю на каждый source × версия). Нужно поддержать фильтрацию `moduleName` для практической применимости.
- **Breaking changes** — отсутствуют, все P3 RPCs дополняют существующие services.
- **Performance** — F6 ListModuleReleases без фильтра может вернуть большой объём; добавим обязательный `module_name` для ограничения.
- **Dependencies** — никаких новых Go зависимостей; используется existing `dynamic.Interface` для unstructured.

## Recommended Direction

**Option A — single batch**. P3 = 4 handler'а небольшой сложности, дублирующие проверенные паттерны P0-P2. Разбиение не оправдано. Один commit, один merge, один CHANGELOG entry «[Unreleased] — P3 — Edge Cases».
